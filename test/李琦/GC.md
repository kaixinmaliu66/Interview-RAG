JS 是自动垃圾回收机制，不需要手动管理内存，核心围绕 **识别不可达对象并释放** 。

### 1. 早期：引用计数（缺点 + 举例）

最早用 **引用计数** ：对象被引用一次计数 + 1，引用消失 - 1，计数为 0 就回收。

**致命问题：循环引用 → 永远无法回收。**

**举例：**

js

```
function foo() {
  let a = {}
  let b = {}
  a.ref = b  // a 引用 b
  b.ref = a  // b 引用 a
}
foo()
```

函数执行完，a、b 已经不可用，但 **互相引用导致计数永远 ≥1** ，无法释放，造成 **内存泄漏** 。

所以现在引擎 **基本废弃引用计数** 。

---

### 2. 现代核心：标记清除（解决循环引用）

现代 V8 核心用 **标记清除算法** ：

1. **标记阶段** ：从 **根对象** （window/global、执行栈、DOM）开始递归遍历， **标记所有可达对象** 。
2. **清除阶段** ：统一清除 **未标记对象** （真正垃圾）。

**为什么能解决循环引用？**

因为循环引用的对象如果 **从根不可达** ，就不会被标记，直接被清除。

---

### 3. V8 分代回收（新生代 From/To + 老生代标记整理）

为提升效率，V8 按**对象生命周期**分两代：

#### ① 新生代（短生命周期，如局部变量）

空间小、回收频繁，用  **Scavenge 复制算法** ：

* 内存平分为  **From** （使用区）和  **To** （空闲区）。
* 分配只在 From。
* GC 时： **存活对象复制到 To** ，然后 **清空 From** ，再 **交换 From/To 角色** 。

**特点：**

* 速度极快、无碎片
* 但空间利用率只有 50%

**为什么适合新生代？**

因为新生代对象 **死得快** ，复制量很小，效率极高。

---

#### ② 老生代（长生命周期：全局、缓存、DOM）

**为什么不用复制算法？**

因为老生代对象多、存活久、体积大， **复制成本极高** ，所以不用 From/To。

**采用：标记清除 + 标记整理**

1. 先**标记清除**垃圾。
2. 再 **标记整理** ：把存活对象 **往一端移动** ，消除内存碎片。

**为什么老生代不改变引用？**

因为老生代对象多、存活稳定， **移动 / 复制代价太大** ，所以只整理、不频繁复制，也不改变对象引用关系，避免大量指针更新开销。

---

### 4. 现代 GC 优化（减少卡顿）

为避免 GC 阻塞主线程，V8 做了：

* **增量标记** ：把标记拆分成小段，和 JS 代码交替执行
* **并发标记** ：后台线程标记，不阻塞主线程
* **惰性清理** ：延迟清理，按需回收

---

### 5. 常见内存泄漏（举例）

* 意外全局变量：`a = 1000` 没声明
* 未清除定时器 / 事件监听
* DOM 引用还在，但节点已删
* 闭包过度持有大对象

---

# 最终一句话总结（面试收尾超稳）

JS GC 从**引用计数**进化到 **标记清除** ，V8 采用 **分代回收** ：

 **新生代用 From/To 复制算法，快但空间减半；老生代用标记清除 + 标记整理，不复制、不改动引用，适合长生命周期对象** ，并通过增量、并发优化降低停顿，同时要避免循环引用与内存泄漏。
